# Patches

Applied to `kubernetes-sigs/vsphere-csi-driver` v3.0.3 during build of
the `vsphere-syncer` binary.

## 001-go-mod.patch

Bumps `go` directive to 1.23 (with toolchain 1.24.2) and refreshes
direct/indirect dependency versions in `go.mod` / `go.sum` to pull in
security fixes and to make the module buildable with the toolchain
declared in our base images.

## 002-fix-cve.patch

Bumps the following direct dependencies in `go.mod` / `go.sum` to
versions that fix CVEs reported by Trivy:

- `google.golang.org/grpc` -> `v1.80.0` (fixes CVE-2026-33186)
- `google.golang.org/protobuf` -> `v1.36.11`
- `golang.org/x/crypto` -> `v0.48.0` (fixes CVE-2025-47914)

The patch was generated by cloning v3.0.3, applying `001-go-mod.patch`,
running `go mod edit -require=...` for the three modules above followed
by `go mod tidy -e` against `GOPROXY=proxy.golang.org`, then
`git format-patch -1`. It only touches `go.mod` / `go.sum`; the build
pipeline then does `rm -rf vendor && go mod download` and the new
versions are pulled fresh from the module proxy.

OpenTelemetry indirect deps (`go.opentelemetry.io/otel*` v0.20.0,
`go.opentelemetry.io/proto/otlp` v0.7.0) are not bumped here: the
`v1.x` line moved/renamed packages still referenced by transitive test
code in `k8s.io/component-base@v0.25` and `k8s.io/apiserver@v0.25`,
so an in-place upgrade is not feasible. The vsphere-syncer binary does
not import any OpenTelemetry package directly, so the residual otel
findings are tracked as `not_affected` in
`images/csi-vsphere-syncer/known_vulnerabilities.vex`.
