# Patches

Applied to `kubernetes-csi/external-provisioner` v6.2.0 (k8s 1.34
track) during build of the `csi-provisioner` binary.

## Delivery model (current)

The k8s 1.34 `csi-external-provisioner` build no longer uses these
patches. `oss.yaml` now points that entry at the Deckhouse fork branch:

```
id: csi-external-provisioner
versions:
  - condition: { k8s: ["1.20"] }   # upstream tag, no VRR
    version: v5.3.0
  - condition: { k8s: ["1.34"] }   # fork branch = v6.2.0 + VRR executor
    version: d8-63742164-vrr
```

`werf.inc.yaml` clones `version` directly from `SOURCE_REPO`
(`.../kubernetes-csi/external-provisioner.git`), so the 1.34 image is
built from branch `d8-63742164-vrr`. The branch already carries the
CVE-fix dependency bumps from `001-fix-cve.patch` (grpc >= v1.80.0,
otel/sdk/metric/trace v1.43.0), so dropping the patch does not regress
CVEs.

Patches under `patches/v6.2.0/` are **NOT applied** for this version:
`werf.inc.yaml` only applies a patch dir whose path contains the
`version` string, and no path contains `d8-63742164-vrr`. They are kept
as a reference / generated diff (see `002-vrr-executor.patch` below).

REMAINING BUILD BLOCKER (unchanged by this switch): the branch imports
`github.com/deckhouse/storage-foundation/api`, and `werf.inc.yaml` still
does `rm -rf vendor && go mod download`. That drops the vendored API and
cannot resolve it over the network. This must be solved by either
(A) publishing `storage-foundation/api` at a real tag/commit and pinning
`go.mod` to it, or (B) building this image `-mod=vendor` without removing
`vendor`. Not resolved here on purpose; do NOT introduce a shadow API.

## 001-fix-cve.patch

Bumps direct/indirect dependencies in `go.mod` / `go.sum` to versions
that fix CVEs reported by Trivy on the v6.2.0 binary:

- `google.golang.org/grpc` `v1.79.1 -> v1.80.0` (fixes CVE-2026-33186)
- `go.opentelemetry.io/otel` `v1.40.0 -> v1.43.0` (fixes CVE-2026-29181)
- `go.opentelemetry.io/otel/sdk` `v1.40.0 -> v1.43.0` (fixes
  CVE-2026-24051, CVE-2026-39883)

Generated by cloning v6.2.0, running `go mod edit -require=...` and
`go mod tidy -e`, then `git format-patch -1`. Only touches `go.mod` /
`go.sum`; the build pipeline does `rm -r vendor && go mod download`
before `make build`. Verified end-to-end inside `golang:1.25-alpine`
with `go build -mod=readonly`.

## 002-vrr-executor.patch

Adds the low-level **VolumeRestoreRequest (VRR) executor** to the
`csi-provisioner` binary. This is the executor half of the VRR feature;
the VRR lifecycle controller (CR lifecycle, ObjectKeeper, TTL,
finalizers, status/conditions) lives in `images/controller` of this
module. The executor only:

- watches `storage.deckhouse.io/v1alpha1 VolumeRestoreRequest` and
  enqueues work on a rate-limited workqueue;
- filters by CSI driver (via the target `StorageClass.provisioner`);
- validates the restore source (`VolumeSnapshotContent` ready-to-use, or
  an existing `PersistentVolume`);
- calls CSI `CreateVolume` (from `volumeSnapshotHandle` or `volumeHandle`);
- creates the target `PersistentVolume` + `PersistentVolumeClaim`
  (explicit binding, no `dataSourceRef`, no temporary `VolumeSnapshot`);
- emits Kubernetes events; is idempotent / retry-safe.

The executor **never writes VRR status** — that is owned by the
storage-foundation VRR controller, which observes the bound target PVC
and sets `Ready`. No `volumerestorerequests/status` RBAC is required for
the provisioner ServiceAccount.

Touches: `cmd/csi-provisioner/csi-provisioner.go` (wiring),
`pkg/controller/vrr_handler.go` (executor), and vendors
`github.com/deckhouse/storage-foundation/api/v1alpha1` (VRR/VCR API
types) with a matching `go.mod` / `vendor/modules.txt` entry.

Source of truth for the executor code is the development branch
`d8-63742164-vrr` in the `external-provisioner` fork; this patch is the
diff of that work against upstream v6.2.0. Verified with
`go build -mod=vendor ./...` and `go test -mod=vendor ./pkg/controller/...`
on a clean v6.2.0 checkout with the patch applied.

### API dependency — temporary vendor-only bootstrap (FOLLOW-UP REQUIRED)

The patch pins `github.com/deckhouse/storage-foundation/api v0.0.0` and
ships the package **only** via `vendor/`. There is no published
tag/commit for that module yet, so it resolves exclusively in vendor
mode. This is a deliberate, temporary bootstrap.

KNOWN CONFLICT: the build pipeline in `werf.inc.yaml` currently runs
`rm -rf vendor` before `go mod download`, which would drop the vendored
API and then fail to resolve `v0.0.0` over the network. Before this
patch can build in the module, one of the following must be done
(tracked as follow-up; do NOT resolve it by copying the API into the
provisioner tree under a new import path — the SSOT is
`storage-foundation/api`):

1. publish `storage-foundation/api` at a real tag/commit reachable via
   `GOPROXY`/`SOURCE_REPO` and replace `v0.0.0` with that version (then
   `rm -rf vendor` is fine); or
2. for this image only, stop removing `vendor` in `werf.inc.yaml` and
   build with `-mod=vendor` so the bootstrap survives.

### RBAC for the provisioner ServiceAccount

In Deckhouse the patched provisioner runs as a sidecar inside each CSI
driver controller Pod (deployed by the driver module, not by this
module), so the executor RBAC must be granted to that driver's
ServiceAccount. The executor needs (NO `volumerestorerequests/status`):

- `volumerestorerequests`: `get`, `list`, `watch`;
- `persistentvolumes`: `get`, `list`, `watch`, `create`, `update`, `patch`;
- `persistentvolumeclaims`: `get`, `list`, `watch`, `create`, `update`, `patch`;
- `volumesnapshotcontents`: `get`, `list`, `watch`;
- `storageclasses`: `get`, `list`, `watch`;
- `secrets`: `get`, `list` (only if the driver uses provisioner secrets);
- `events`: `create`, `patch`, `update`.

A ready-to-apply additive manifest and helper live at
`hack/vrr-rbac-manual.yaml` and `hack/apply-vrr-rbac.sh`. Most of these
verbs already exist in a driver's provisioner ClusterRole for normal
provisioning; the only strictly net-new permission is
`volumerestorerequests` `get/list/watch`. FOLLOW-UP: fold this into the
shared CSI-driver helm lib so every driver gets it automatically.
